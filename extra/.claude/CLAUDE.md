## テストの実装方法

1. 抑えておきたいテストシナリオを洗い出し、リスト（テストリスト）にまとめる。この段階でレビューをもらう
2. テストリストの中から「ひとつだけ」選び出し、実際に、具体的で、実行可能なテストコードに翻訳し、テストが失敗することを確認する
3. プロダクトコードを変更し、今書いたテスト（及び、それまでに書いたすべてのテスト）を成功させる（その過程で気づいたことはテストリストに追加する）
4. 必要に応じてリファクタリングを行い、実装の設計を改善する
5. テストリストが空になるまで `2` に戻って繰り返す

テストコードを書くときは Given, When, Then パターンで記述する。
typescript だと以下のようなスタイルで記述する。
Given/When/Then の間に一行空行を入れることで可読性を担保する

```typescript
describe ('テスト対象について明確に記述する', () => {
  it('なにをテストするのか明確に記述する', async () => {
    // Given: 前提条件について明確に記述
    const input = { key: 'value' };

    // When; どんなアクションをするのか明確に記述する
    const result = await logic(input);

    // Then: 期待する出力について明確に記述する
    expect(result).toEqual({ status: 'success' });
  });
})
```

pytest や go など parametrized test / table driven test の場合、Given/When となる部分はパラメータのコメントに明確に記述する

## バージョン管理

## Allow

- `jj log` : 履歴確認
- `jj st` : 変更点確認
- `jj diff` : 差分確認
- `jj desc` : 説明の編集
- `jj show` : 変更内容表示
- `jj check` : pre-commit チェック

## Deny

以下の操作はユーザーが実行するため禁止されています。

- `jj commit`
- `jj git push` : リモートへのプッシュ

## Plan の作成フロー

- 実行計画を作成する際に暗黙的においている仮定について Plan の先頭に詳細に記述する。この仮定が間違っている場合は Plan そのものが破綻するので大規模な計画の場合は事前に暗黙的な仮定があっているか確認する
- 実行計画を Plan Mode で作成する場合、1 門 1 答形式で必要な情報をユーザーに訪ね、システム要求を精緻化してください
- 細かい単位でユーザーにコードレビューを求めるか、まとめて完成させてから報告するか選択させてください。この回答によって jj commit の粒度を変える必要があります。

### 細かい単位でコードレビューを求める場合の実装計画

こちらを要求される場合、大きな新機能の追加や複雑な開発タスクを実行することとなります。

レビューの容易性、保守性のため commit の粒度を細かくする必要があります。

1. README.md 等 Docs の作成、関数のインタフェース定義、テストリストの定義ができた時点でレビューを実施する。jj-desc / jj new でこの時点の作業を記録する
2. 1 つずつ関数の実装とテストを行い、レビューを実施する。jj-desc / jj new でこの時点の作業をきろくする
3. `2` を繰り返し、細かい粒度の Commit を重ねていき完成させる

### まとめて完成させる場合

Plan ファイルに則って開発を完了させてからレビューを依頼する。jj-desc / jj new で作業を記録する

## ソースコードの分析

実装計画や調査の際に必要となるライブラリ等のコードの調査は Websearch ではなく @~/ghq/ 配下に Clone してある実際のソースコードを参照する。Clone されていない場合は Clone してくるようユーザーに依頼する
